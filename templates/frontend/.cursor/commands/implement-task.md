# /implement-task（frontend）

目标：以最小变更完成页面/API 需求，保证可回归与可交付。

## 输入
- 任务目标（页面、交互、接口）
- 相关 `openapi.yaml` 或接口约定
- 现有组件与路由约束

## 执行步骤
1. 先 `/api-search`，确认是否可复用已有接口与组件。
2. 先定 API 影响，再改页面与状态逻辑。
3. 保持目录落点一致（views/components/api/router/store）。
4. 补齐最小验证（lint/tsc/关键交互自测）。
5. 产出 PR 说明草稿（复用依据、契约影响、验证结果）。

## 输出格式（必须）
- 按 `PR_TEMPLATE.md` 输出 **2)~5)** 对应字段，标题保持一致：
  - **2) 变更范围（必填）**
    - `业务目标（1-2 句）`
    - `代码改动范围（views/components/api/router/store）`
    - `是否新增或调整路由：是/否`
    - `是否变更权限守卫：是/否`
  - **3) 契约影响（必填）**
    - `受影响接口（逐条）`
    - `契约结论：有变更 / 无变更`
    - `是否更新 openapi.yaml：是/否`
    - `若“无变更”，说明依据`
    - `前后端兼容性评估`
  - **4) 验证证据（必填）**
    - `本地验证命令`
    - `页面验证清单（至少 2 条）`
    - `关键截图/录屏说明（如适用）`
  - **5) 风险与回滚（必填）**
    - `主要风险`
    - `监控与观察点`
    - `回滚方案`

## 复制规则（必须）
- `/api-search` 的结果用于填充 PR 的 **1) 复用依据（必填）**。
- 本命令输出用于填充 PR 的 **2)~5)**。
- 输出末尾附上 **6) 提交前自检（勾选）** 的勾选建议（仅勾选已满足项）。
